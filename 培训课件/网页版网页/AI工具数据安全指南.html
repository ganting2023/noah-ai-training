AI 工具数据安全指南

在金融、法务、合规、风控场景里，AI 不是“能不能提高效率”那么简单，而是“这份数据能不能出机房、能不能出本机、能不能出给第三方”。这页不讲抽象原则，只讲真正会遇到的判断和动作。

首次阅读约 8–12 分钟 建议在正式使用 AI 前完整看一遍 判断 → 工具 → 分级 → 脱敏 → 本地模型
先看判断框架 打开安装指南
最短路径
先判断后操作
01
先分数据等级 公开信息、内部一般信息、客户敏感信息、监管级机密，处理方式完全不同。
02
再选使用方式 网页版、API 模式、本地模型，不只是体验差异，更是数据边界差异。
03
最后做脱敏和隔离 用 `.claudeignore`、`.cursorignore`、沙盒和占位符，别把风险留给习惯。
第一判断
能不能外发
不是所有 AI 问题都该先想“哪个模型更强”，先想“这份内容能不能离开本机”。
优先路径
API 模式
在企业账户和正确配置下，Claude Code、Codex CLI、Gemini CLI 比网页聊天更可控。

客户 PII / 审计材料
含真实姓名、证件号、账号、调查材料时，不要默认丢给云端模型。
兜底方案
本地模型
Ollama 不是万能，但对“不能外发”的内容，至少给了一个现实的可用选项。

先问 3 个问题

大多数误用不是出在不会配工具，而是出在一开始就没判断清楚这份数据属于什么。先把这三问走完，再决定用哪个工具。

先看这一节

1. 这份内容是否包含真实客户信息？

  • 姓名、身份证、护照号、手机号、银行卡、账户余额、KYC 材料，都算。
  • 只要有一项是真实可识别信息，就不要默认直接发云端。

2. 这份内容是否属于监管级或调查级材料？

  • 内部审计报告、处罚调查、未公开风险事件、监管沟通材料，默认高敏。
  • 这类内容通常更适合本地模型，或者干脆不用 AI。

3. 这次 AI 需要做什么？

  • 只是提纲、归类、格式化，还是需要完整读文件、改文件、跑命令？
  • 任务越深，越要先收权限范围，再让 AI 开始处理。
⚠️
底线：如果你都说不清这份内容能不能外发，就先不要把它交给云端 AI。先确认数据等级，再决定是脱敏、改走 API，还是改成本地模型。
这份数据要给 AI 处理？ ↓ 包含真实客户 PII（姓名 / 证件 / 账户）？ ├── 是 → 先脱敏，或改用 Ollama 本地模型 └── 否 → 属于监管级机密（调查 / 审计 / 未公开事件）？ ├── 是 → 用本地模型，或不用 AI └── 否 → 走 API 模式（Claude Code / Codex CLI / Gemini CLI） ↓ 确认 ignore 文件与权限范围 ↓ 使用企业账户 / 企业合同 ↓ ✅ 可以开始操作
📡

你的内容发到哪里了？

很多人以为“我只是粘了一段文本”不算外发。实际上只要用了云端服务，内容就已经发到了对应供应商的服务器。区别只在于训练政策、账户类型和隔离程度。

先搞清数据去向
工具 数据发送到 是否可能用于训练 推荐做法
网页版 Claude（免费 / Pro） Anthropic 服务器 视设置与账户策略而定 关闭“改进产品”类选项，不上传敏感文件
Claude Code（API 模式） Anthropic API 默认不用于训练 优先使用企业账户，配 `.claudeignore`
网页版 ChatGPT OpenAI 服务器 需手动关闭相关数据开关 只适合非敏感文本问答
Codex CLI（API 模式） OpenAI API 默认不用于训练 优先企业账户，敏感操作先预演
Cursor Cursor + 选用的 AI 提供商 视 Privacy Mode 与供应商模式而定 开启 Privacy Mode，避免打开敏感文件
Gemini CLI（企业 API） Google Cloud 企业 API 不用于训练 必须走企业 Google 账户，不要混用免费层
Ollama（本地模型） 本机 不外发 适合不能出网的数据，但能力弱于顶级云模型
💡
关键结论：如果你必须用云端，优先走 API 模式 而不是网页聊天。它的优势不是“更技术”，而是账户策略更清楚、训练政策更明确、权限和上下文隔离更可控。
🛠

各工具的实操红线

同样是“让 AI 处理文件”，不同工具的默认行为差别很大。这里不讲功能，只讲你真正该盯住的几条红线。

配置比口头提醒更重要

Claude Code

  • 先看权限提示，再授权，不要图快一路确认。
  • 在工作目录里放 `.claudeignore`，把客户原始数据、凭证、未脱敏文件先挡掉。
  • 不要用 `--dangerously-skip-permissions` 这类绕过检查的方式。
  • 法务 / 合规场景下，先限定工作目录，再开 Claude。
# .claudeignore 示例
/客户原始数据/
**/*未脱敏*
/内部账户信息/
.env
secrets/

Codex CLI

  • 优先从受限模式开始，先让 AI 说“准备怎么做”，再放开执行。
  • 敏感任务先做“建议模式 / 干跑预演”，不要直接让它自动跑。
  • 提示词里不要直接放 API Key、账号密码、证件号。
  • 高敏材料不要混在当前工作目录里一起打开。

Cursor

  • 开启 Privacy Mode，再开始接触真实业务文件。
  • 当前打开的文件会被补全和上下文读取，敏感文件不要“顺手点开”。
  • 用 `.cursorignore` 先屏蔽数据目录、KYC 文件、密钥文件。
# .cursorignore 示例
/客户数据/
/KYC文件/
*.pem
*.env
/内部报告/未脱敏/

Gemini CLI / 网页版 AI

  • Gemini CLI 只建议用企业 Google 账户，不要把免费层当企业工具用。
  • 网页版 Claude / ChatGPT 更适合非敏感问答，不适合上传真实客户材料。
  • “只是传一页 PDF 让它看看”也算外发，不要被交互形式误导。
📂

敏感数据分级使用原则

不要把“敏感不敏感”只理解成带不带姓名。真正好用的方式，是把常见数据分成四档，每档对应固定处理动作。

先分级再操作
数据级别 常见例子 推荐做法
公开信息 监管官网公告、公开基金说明书、公开新闻稿 可直接交给任意 AI 工具处理
内部一般信息 内部流程文档、非客户模板、通用培训材料 优先用 API 模式，并配置 ignore / 权限范围
客户敏感信息 真实姓名、证件号、手机号、账号、余额、KYC 文件 必须先脱敏，再提交；或改用本地模型
监管级机密 处罚调查、内部审计报告、未公开风险事件、监管沟通记录 优先本地模型；不能明确控边界时，宁可不用 AI

可以大胆用 AI 的内容

公开新闻、公开法规、公开说明书、内部通用模板、培训讲义、无真实客户识别信息的草稿，通常都适合直接用 AI 提高效率。

不要抱侥幸心理的内容

“只是先让 AI 帮我看一下”的客户材料、包含账号或证件的截图、内部调查报告、带项目名单的 Excel，都不该默认往云端丢。

🎭

脱敏操作：两分钟学会

“脱敏”不是删光信息，而是把可识别字段替换成占位符，让 AI 还能完成分析，但拿不到真实身份信息。

最实用的日常动作

❌ 原文（不能直接提交）

客户张三，身份证 440301199001011234，账户余额 1,200,000 港元，联系电话 138xxxx8888。

✅ 脱敏后（可以提交）

客户 [客户A]，身份证 [证件号已核验]，账户余额 [X金额] 港元，联系电话 [联系方式已脱敏]。

🧩
核心原则：让 AI 保留结构和关系，但看不到真实标识。它仍然能做分类、比对、摘要、找不一致，但无法还原出具体客户。
请扫描当前目录中的所有 .docx 和 .pdf 文件，
识别并替换以下敏感字段（不删除，用占位符替换）：
- 真实人名 → [姓名已脱敏]
- 身份证 / 护照号 → [证件号已脱敏]
- 银行账号 → [账号已脱敏]
- 手机号 → [联系方式已脱敏]

要求：
1. 保留文档原有结构，不要删除段落
2. 输出一份脱敏报告，说明每份文件替换了哪些字段以及数量
3. 另存为新的脱敏版本，不要覆盖原文件
🖥️

本地模型与安全边界

Ollama 不是为了“更酷”，而是为了给不能外发的数据一个现实可行的落点。它不一定最强，但它让数据不离开本机。

高敏场景的重要选项

☁️ 云端模式

你输入内容 → 加密发到 Anthropic / OpenAI / Google 服务器 → 模型处理 → 返回结果。优点是能力强，缺点是数据离开本机。

🖥️ 本地模式

你输入内容 → 本机 CPU / GPU 运算 → 本机直接返回结果。优点是数据零出境，缺点是能力和速度取决于硬件。

电脑配置 推荐模型 能力水平 适合场景
普通笔记本（8GB 内存） qwen2.5:3b / llama3.2:3b 基础水平 文档核查、关键词提取
16GB 内存 / 独显 qwen2.5:14b / llama3.1:8b 推荐档 多数法务文档与脱敏任务
32GB 内存 / 高端显卡 qwen2.5:32b / llama3.3:70b 接近高端云模型 复杂文件分析、长文档处理

本地模型适合什么

  • 不能出网的材料做初步整理、脱敏、摘要。
  • 内部机密内容做第一轮结构化提取。
  • 作为云端模型前的安全缓冲层。

本地模型的现实局限

  • 综合推理能力仍普遍弱于顶级云模型。
  • 速度和可用性受本机硬件影响很大。
  • 不适合把所有复杂任务都完全替代云端。
💡
最实用的组合：敏感数据先用 Ollama 做初步整理和脱敏，脱敏后的文本再交给 Claude Code 或 Codex CLI 做深度分析。安全和能力不是二选一，可以分阶段组合。
# 1. 安装 Ollama
# https://ollama.com

# 2. 下载中文优化模型（推荐）
ollama pull qwen2.5:14b

# 3. 启动并测试
ollama run qwen2.5:14b

每次使用前后的检查清单

真正防风险的不是“我知道原则”，而是每次操作前后都重复同一个检查动作。把它变成惯例，比任何口号都有效。

建议打印 / 固化到流程

操作前

  • 我是否明确知道这份数据的等级？
  • 当前工具是网页版、API 模式，还是本地模型？
  • 工作目录里有没有不该被读取的文件？
  • ignore 文件和权限范围是否已经配置？
  • 若含客户信息，是否已经脱敏？

操作后

  • 生成结果里是否残留真实客户信息？
  • 导出的文件是否误带原始附件或截图？
  • 是否需要把脱敏版和原始版分开存放？
  • 这次 AI 做了哪些关键操作，是否要记录？
  • 下次同类任务是否该沉淀成固定模板？
🔒
最后一句：在高敏业务里，AI 首先是一个需要被管控的外部处理能力，其次才是一个效率工具。把边界定清楚，才有资格谈提效。
